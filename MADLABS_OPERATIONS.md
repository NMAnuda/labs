# MADLABS Operations & Engineering Constitution

This document is the Single Source of Truth (SSOT) for all developers and AI agents working on MADLABS projects. It summarizes the core mandates from the central `MADLABS-operations` repository.

---

## 1. Engineering Standards ("Clean, Typed, and Tested")

Every MADLABS project must adhere to these foundational principles:

### Core Mandates
- **Plan-First Workflow:** No implementation starts without an approved plan and Gherkin-style Acceptance Criteria.
- **Type Safety:** `noImplicitAny` is non-negotiable. Use **Zod** (TypeScript) or **FluentValidation** (.NET) for all runtime validation.
- **Functional Preference:** Prefer functional programming patterns (const functions) over class structures in TypeScript.
- **TDD (Test Driven Development):** Write the test, watch it fail, implement, watch it pass. No code is merged without verification.
- **Multi-Tenancy:** Data isolation (scoping by `companyId` or `tenantId`) must be enforced at the lowest possible level (ORM filters or middleware).
- **Financial Precision:** Never use `float` or `number` for currency. Use `Decimal.js` or standard `decimal` types.

### Standard API Contract
All APIs must return this structure:
- **Success:** `{ "data": { ... }, "meta": { "total": 100 } }`
- **Error:** `{ "error": { "code": "STRING", "message": "Human readable", "details": [] } }`

## 2. The Life of a Project (Launch Cycle)

1. **Phase 1: Conceptualization:** Define business value and feasibility.
2. **Phase 2: Architecture & Design:** Create PRD/TDD and multi-sprint roadmap.
3. **Phase 3: Provisioning:** Setup repo using `madlabs-master-template` and configure GCP IaC.
4. **Phase 4: Sprint 0:** Achieve "Hello World" in production (Auth, Theme, DB).
5. **Phase 5: MVP Iteration:** Build core features using the **Life of a Task** workflow.
6. **Phase 6: Launch & Handoff:** Security/Performance audits and production cutover.

---

## 3. The Life of a Task (Execution Cycle)

All work items must move through these stages on the GitHub Delivery Board:

1. **🟢 Discovery (Backlog):** BA creates issue with **Gherkin AC** (Given/When/Then).
2. **🟡 Refinement (Ready):** Tech Lead approves the Technical Plan in the comments.
3. **🔵 Execution (In Progress):** Developer builds features using TDD. Links PR to issue.
4. **🟣 Verification (QA / UAT):** Senior QA verifies AC in the Staging environment.
5. **⚪ Operations (Done):** PM Intern archives and updates stakeholders.

---

## 4. Security & Deployment Gates

- **Production Gatekeeper:** Direct merges to `main` are restricted. The `deploy-prod` job in GitHub Actions must check the `github.actor` to ensure only the CTO or Senior DevOps can trigger production releases.
- **Secrets Management:** NEVER commit secrets to source control. Use GCP Secret Manager for production and `.env.example` as a template for local development.

---

## 5. AI-Assisted Development & Governance

- **Human-in-the-Loop:** AI agents (Copilot, Cursor, Gemini) are tools, not authors. A human developer is 100% accountable for every line of code committed.
- **Verification Mandate:** AI-generated code MUST be manually verified against these standards and pass all TDD requirements before merging.
- **Credential Safety:** NEVER paste production secrets or PII into AI chat interfaces.
- **IP Ownership:** All code generated by AI agents within this workspace is the proprietary property of MADLABS.lk as per the `LICENSE`.

---

## 6. Definition of Done (DoD)

A task is complete ONLY when:
1. [ ] Code follows **Clean Architecture** patterns.
2. [ ] **Validation** schemas (Zod) are implemented.
3. [ ] **TDD Tests** (Unit/Integration) are passing in CI.
4. [ ] **Acceptance Criteria** are verified.
5. [ ] **Documentation** (JSDoc/Markdown) is updated.
6. [ ] **Lint & Typecheck** pass without warnings.

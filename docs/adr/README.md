# Architecture Decision Records (ADRs)

Record important architectural decisions made during the project lifecycle.

## What is an ADR?

An Architecture Decision Record captures a single architectural decision along with its context and consequences. ADRs help the team understand **why** decisions were made, not just **what** was decided.

## How to Use

1. Copy `000_ADR_Template.md` and rename it with the next sequential number (e.g., `001_Use_PostgreSQL.md`)
2. Fill in all sections of the template
3. Submit the ADR as part of a Pull Request for team review
4. Once accepted, the ADR is considered **Accepted** and should not be modified (supersede it with a new ADR instead)

## ADR Status Lifecycle

| Status      | Meaning |
|-------------|---------|
| Proposed    | Under discussion, not yet decided |
| Accepted    | Decision has been agreed upon |
| Deprecated  | No longer relevant but kept for historical context |
| Superseded  | Replaced by a newer ADR (link to replacement) |
